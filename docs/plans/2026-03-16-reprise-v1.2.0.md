### Regles git pour les agents

1. **Chaque agent cree sa branche depuis `dev`** (apres que dev soit propre post-T6)
2. **Chaque agent commit son travail sur SA branche** — jamais directement sur dev
3. **Les merges vers dev passent par l'Orchestrateur** apres QA + review + GO user
4. **Pas de force push** — si conflit, l'agent signale et attend instruction
5. **Convention de nommage** : `<type>/<scope>` (chore/, fix/, feat/, release/)
6. **Message de commit** : `<type>(<scope>): <description>` (ex: `feat(v1.2.0): traduction descriptions WEAPON`)
7. **Un agent qui a fini sa branche ne touche plus rien** — c'est le Reviewer puis l'Orchestrateur qui decident

### Parallelisme et isolation

Les 4 agents desc-* travaillent **chacun dans un git worktree** isole :

```bash
# L'Orchestrateur cree les worktrees avant de lancer les agents
cd "D:/Fractal Softworks/Starsector/mods/starsector_lang_pack_fr_private"
git worktree add ../worktrees/desc-weapon feat/v1.2.0-desc-weapon
git worktree add ../worktrees/desc-ship feat/v1.2.0-desc-ship
git worktree add ../worktrees/desc-system feat/v1.2.0-desc-system
git worktree add ../worktrees/desc-autres feat/v1.2.0-desc-autres
```

Chaque agent travaille dans son worktree, sur son fichier CSV partiel. L'assembleur recompose le CSV final.

### Avantages de cette approche git

- **Isolation totale** : un agent qui deraille ne casse pas le travail des autres
- **Rollback facile** : chaque branche peut etre abandonnee sans impact
- **Tracabilite** : `git log --oneline --graph --all` montre exactement qui a fait quoi
- **Review naturelle** : chaque merge = point de review (comme une PR)
- **Parallelisme reel** : worktrees = pas de contention sur le repo

### Inconvenients

- **Overhead git** : creation/suppression de branches et worktrees
- **Assemblage CSV** : le merge des 4 fichiers partiels en un seul demande un agent dedie
- **Complexite** : plus de branches = plus de merges potentiellement conflictuels (mitigation : chaque agent travaille sur des lignes differentes)

## Gates de review obligatoires

Les reviews sont des **points de blocage** : le user doit approuver avant de continuer. Elles sont executees via l'agent `superpowers:requesting-code-review` ou `superpowers:code-reviewer`.

| # | Apres tache | Scope de la review | Reviewer |
|---|-------------|-------------------|----------|
| REV1 | Task 3+4 (nettoyage) | Diff git avant merge : rien de precieux supprime, gitignore correct, structure propre | Agent code-reviewer |
| REV2 | Task 5+6 (merge dev) | Etat des branches : merge propre, pas de conflit residuel, GitHub synchro | Agent code-reviewer |
| REV3 | Task 8 (tests consolides) | Suite de tests : couverture, chemins corrects, pas de tests desactives | Agent code-reviewer |
| REV4 | Task 10 (strings.json traduit) | **Review traduction** : qualite FR, typographie, variables preservees, coherence terminologique | Agent code-reviewer + Biblium |
| REV5 | Task 11 (descriptions.csv traduit) | **Review traduction** : qualite FR, pas de texte EN residuel, glossaire respecte, CSV intact | Agent code-reviewer + Biblium |
| REV6 | Task 12 (packaging) | mod_info.json, README, contenu ZIP, rien d'oublie | Agent code-reviewer |
| REV7 | Task 13 (pre-release) | **Review finale** : diff complet public vs privé, release notes, pas de fichiers sensibles | Agent code-reviewer |

### Protocole de review

1. **L'agent executant termine sa tache + checkpoint QA**
2. **L'agent code-reviewer est lance** avec le scope specifique
3. **Le reviewer produit un rapport** : PASS / PASS_WITH_COMMENTS / FAIL
4. **Si FAIL** : retour a l'agent executant avec les corrections demandees, puis re-review
5. **Si PASS** : le user est notifie et donne son GO pour la tache suivante
6. **Le user a toujours le dernier mot** — un PASS du reviewer ne suffit pas sans GO explicite du user

### Reviews de traduction (REV4 + REV5) — checklist specifique

L'agent code-reviewer doit verifier :
- [ ] Pas de texte EN residuel dans les fichiers traduits
- [ ] Variables `$varName`, `{0}`, `\n` preservees intactes
- [ ] Typographie FR : espaces insecables avant `:;!?`, guillemets `<< >>`, points de suspension `...`
- [ ] Noms propres non traduits (factions, vaisseaux specifiques)
- [ ] Coherence avec le glossaire Biblium (`biblium.search` pour verification spot)
- [ ] Ton coherent avec les traductions existantes (tips, tooltips)
- [ ] Pas de troncature de texte (longueur comparable a l'original)
